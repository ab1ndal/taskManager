# Phase 2.5: Task List Corrections & Enhancements

**Date:** 2026-03-30
**Status:** Approved
**Scope:** 13 corrections and enhancements to the task list before Phase 3 begins

---

## Overview

Phase 2 shipped working task creation with optimistic UI, but revealed 13 issues during review. This phase addresses them as a dedicated "Phase 2.5" inserted between Phase 2 and Phase 3 in the roadmap. Work is grouped into four plans.

---

## Plan 1: State Sync & Bug Fixes (items 1, 2, 3)

### Root Cause

`TasksPageClient` initializes `localTasks` with `useState(initialTasks)`. After `revalidatePath` triggers a server re-render, the new `initialTasks` prop arrives but `useState` ignores it — state is stale. This causes:

- Optimistic tasks staying grey permanently (item 1)
- Completed tasks staying in the active list (item 2)
- Deleted tasks staying in the active list (item 3)

### Fix

Add a `useEffect` in `TasksPageClient` that watches `initialTasks`. When it changes, replace `localTasks` with the new server data and clear `optimisticTaskIds`. No new server actions needed — `completeTask` and `deleteTask` already exist and call `revalidatePath`.

```ts
useEffect(() => {
  setLocalTasks(initialTasks);
  setOptimisticTaskIds(new Set());
}, [initialTasks]);
```

Complete and delete buttons already exist in `TaskCard` and provide `pending` opacity during the transition.

## Plan 2: Date & Bucketing Logic (items 5, 7, 8, 9)

### Due Date Input

Change `type="datetime-local"` to `type="date"` in `NewTaskModal`. Store as `YYYY-MM-DDT00:00:00Z` in the DB. Subtask rows already use `type="date"` — this makes the parent consistent.

### Bucketing Rules

Replace time-based comparison in `bucket-tasks.ts` with calendar date comparison:

| Bucket | Rule |
|--------|------|
| Overdue | `due_at` date < today |
| Today | `due_at` date = today |
| Upcoming | `due_at` date > today, or no `due_at` |

Deadline labels:
- Overdue → "Overdue" (red)
- Today → "Due today" (yellow)
- Upcoming → "Due in X days" (green)

Hours-based label removed entirely.

## Plan 3: Task Enhancements (items 4, 10, 11, 12)

### Edit Modal (item 4)

New `EditTaskModal` component, pre-filled with current task values. Same layout and field structure as `NewTaskModal`. Fields: title, description, due date, assignees.

New `updateTask` server action:
```ts
updateTask({ taskId, title, description, dueAt, memberIds })
```
Updates `tasks` table fields and reconciles `task_assignments` (add/remove rows as needed). Edit is optimistic — card updates immediately and reverts on error. Edit icon button added to each active `TaskCard`.

### Subtask Description (item 12)

Add `description` textarea to each subtask row in `NewTaskModal`. The `tasks.description` column already exists. Add `description?: string` to `SubtaskRow` type and pass through `createTaskWithSubtasks`.

### Subtasks Displayed Inline (item 10)

`RawTask` gains a `subtasks` field:
```ts
subtasks?: { id: string; title: string; completed_at: string | null }[]
```

Server query in `page.tsx` fetches subtasks via join or separate query. Parent task cards render subtasks as a nested list below the title, expanded by default. Each subtask row has its own complete button.

### Auto-Complete Parent (item 11)

In `completeTask` server action, after marking a subtask complete:
1. Query sibling subtasks: `tasks WHERE parent_task_id = X AND completed_at IS NULL`
2. If count = 0, mark parent complete as well

A task with subtasks is not completable directly from its card — only via completing all subtasks.

## Plan 4: Drag-to-Reorder & Recurring Tasks (items 6, 13)

### Drag-to-Reorder (item 6)

**Library:** `@hello-pangea/dnd` (maintained react-beautiful-dnd fork, no Framer Motion, Next.js compatible)

- Each bucket section (`Overdue`, `Today`, `Upcoming`) is a `Droppable`
- Each task card is a `Draggable`
- Full card is the drag handle; cursor changes to `grab` on hover, `grabbing` on drag
- Dragging between buckets is disabled (same-bucket reorder only)

**Sort key update:** New `reorderTask` server action computes a midpoint `member_sort_key` between the task's new neighbours:
```ts
newKey = (prevKey + nextKey) / 2
```
If the gap becomes too small (< 1), a full renumber of the bucket is triggered. Reorder is per-user — shared tasks reorder independently for each assignee.

### Recurring Tasks (item 13)

**Creation UI:** In `NewTaskModal`, add a "Repeat" dropdown below the due date:
- `Does not repeat` (default)
- `Repeats`

When `Repeats` is selected, show: "Every ___ days" number input (min 1).

**Storage:** Creating a recurring task inserts a `task_rules` row:
```ts
{ workspace_id, title, interval_count: frequencyDays, frequency: 'days', next_run_at: due_at, is_active: true }
```
The task gets `rule_id` set to this rule's id.

**Completion behaviour:** In `completeTask`, if the task has a `rule_id`:
1. Compute `next_due = due_at + interval_count days`
2. Create a new task instance (same title, workspace, assignees, rule_id)
3. Update `task_rules.next_run_at = next_due`
4. Delete the completed task — it never appears in the completed list

`RawTask` gains `rule_id?: string` so the client can show a repeat indicator (loop icon) on the card.
