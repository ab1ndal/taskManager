## Modal Behavior

### Entrance & Exit
- **Entrance:** Fade in with scale animation (0.97 → 1 over 200ms) defined in `@keyframes modal-in` in globals.css
- **Exit:** Reverse animation (1 → 0.97, opacity 1 → 0) over 150ms (on close or successful submission)
- **Backdrop:** Semi-transparent black overlay (black/40) dismisses modal on click

### Form Interaction
- **Auto-focus:** Title input receives focus on modal open
- **Submission blocked:** When title is empty, no assignee selected, or request is pending
- **Workspace switch:** Resets assignee selection to creator's member ID in new workspace (decision from CONTEXT.md)
- **Assignee default:** Creator is pre-selected when modal opens (decision from CONTEXT.md)
- **Subtask management:** Press Enter in subtask title to add new row; click ✕ to remove

### Loading State
- **Button text changes:** "Add task" → "Adding…" while `pending` is true
- **Button disabled:** All form fields + buttons disabled during submission (`disabled={pending}`)
- **Field opacity:** Disabled fields use `opacity-50` class
- **No spinner on button:** Loading text alone sufficient

### Feedback Flow

**On successful submission (no subtask errors):**
1. Optimistic task added to list with `opacity-40` class (40% opacity)
2. Modal closes immediately
3. Form resets (all fields cleared, creator re-selected as assignee)
4. Success toast fires: "Task created"
5. Server confirms → optimistic task opacity returns to 100%

**On successful submission (with subtask partial failure):**
1. Parent task created optimistically, subtasks that failed are omitted
2. Modal closes immediately
3. Warning toast fires: "Task created, but N subtask(s) could not be saved"
4. No success toast in this path (warning toast covers it)

**On error:**
1. Optimistic task is removed from list
2. Modal remains open (does NOT auto-close)
3. Error toast fires: "Failed to create task"
4. User can correct and resubmit, or cancel

### Optimistic Task Styling
- **Container:** Same TaskCard structure as confirmed tasks
- **Opacity:** `opacity-40` class while `pending` in parent state
- **No special badges:** Pending task shows no "pending" indicator; opacity alone signals unconfirmed state
- **Smart bucket placement:** Client-side logic mirrors `bucket-tasks.ts` to place optimistic task in correct bucket (Today/Upcoming/Overdue) based on due date
- **Visibility:** Only visible to creator until server confirms (optimistic state lives in parent's local state)

## Design Notes

### Optimistic Feedback Philosophy
This phase prioritizes instant visual feedback over confirmation delays. The optimistic task appears immediately in the correct bucket (Today/Upcoming/Overdue) via smart client-side bucketing logic that mirrors the server. The task becomes fully opaque when the server confirms, providing a subtle "refresh" that assures the user the operation succeeded.

### Assignee Pre-selection
Defaulting the creator as assignee solves two problems: (1) ensures the task is visible to someone, and (2) provides a sensible default that matches user intent 80% of the time. Submit button blocks if no assignee is selected, preventing invisible tasks from being created.

### Subtask Partial Failure Handling
If some subtasks fail to save, the warning toast ("Task created, but N subtask(s) could not be saved") informs the user of the partial failure while keeping the primary task creation flow unblocked. This is a pragmatic compromise: the parent task is always saved, and the user can add missing subtasks via task detail (Phase 3).

### Modal Focus Management
Auto-focusing the title field on open follows web form convention and reduces friction. Workspace switching resets assignees to the creator's member ID in that workspace, preventing the user from accidentally creating an unassigned task when switching workspaces mid-form.
